정량적 위험평가 (SLE·ALE·ROI)

13~21 확장 신규
16회 실무19회 실무21회 실무이후 11회 공백 → 임박도 4.11
핵심 공식 — 대입값·단위까지 써야 만점
  • SLE = AV × EF (단일손실 = 자산가치 × 노출계수)
  • ALE = SLE × ARO (연간손실 = 단일손실 × 연간발생빈도)
  • 대책 효과(가치) = 감소된 ALE − 대책 운영비용
  • ROI(%) = (ALE − X) / X × 100 (X = 위험 제거 투입비용)
📚 기출 · 16회 15번(실무)
정보보호 위험평가에서 AV=100,000, EF=0.2, ARO=0.5 일 때 SLE·ALE를 구하고, 감소 ALE=30,000·운영비=17,000 일 때 대책 효과를 구하시오.
모범답안 보기
SLE = 100,000 × 0.2 = 20,000 ALE = 20,000 × 0.5 = 10,000 대책 효과 = 30,000 − 17,000 = 13,000
GICHUL 16-15 · 21-15에 재출제(7회 문제 그대로)
📝 32회 예상문제
어떤 자산의 AV=2,000,000, EF=0.5, ARO=0.2 이다. 1) SLE와 ALE를 계산식과 함께 구하시오. 2) 연 손실을 완전히 없애는 보안대책에 비용 X=300,000을 투입했다. ROI(%)를 구하시오.
모범답안 보기
1) SLE = AV×EF = 2,000,000 × 0.5 = 1,000,000 ALE = SLE×ARO = 1,000,000 × 0.2 = 200,000 2) ROI = (ALE − X)/X × 100 = (200,000 − 300,000)/300,000 × 100 = −33.3% → ROI가 음수이므로 대책 비용이 기대 손실보다 커 비효율적. 대책 재검토 필요.
함정 · 공식만 쓰고 대입 과정을 생략하면 부분점수. ROI가 음수로 나오면 "비효율" 해석까지 붙여야 완결.

DDoS·DRDoS 증폭

18회20회22회23회27회gap 5 · n5 최고 신뢰도
원리 · 대응
  • 원리: 출발지 IP를 피해자로 스푸핑 → 반사서버에 소량 질의 → 대용량 응답이 피해자로 증폭
  • 증폭 벡터: DNS ANY 질의 · NTP monlist · memcached
  • 차이점: 반사구조라 출처 은닉 + 좀비 없이도 대량 트래픽 생성
  • 대응: Open Resolver 제거 · uRPF(출발지 검증) · RRL(응답률 제한) · monlist 비활성화(disable monitor)
📚 기출 · 18회 12번(서술)
DRDoS에 대하여 설명하시오. 1) 공격 원리 2) 기존 DoS와의 차이점 3) Unicast RPF
모범답안 보기
1) 소스 IP를 공격대상 IP로 위조해 다수 반사서버에 요청 → 대상은 반사서버의 응답 폭주로 서비스 거부 2) ① 출발지 위조+반사서버 경유로 출처 파악 곤란 ② 좀비 없이 증폭으로 고효율 3) 인터페이스로 들어온 패킷의 IP를 라우팅 테이블로 역검증, 경로가 맞지 않으면 위조로 보고 폐기
GICHUL 18-12 · 22-16(DNS ANY 증폭) · 27-18(NTP)
📝 32회 예상문제
NTP 서비스를 악용한 DRDoS 공격에 대하여, 1) 어떤 기능이 악용되는지와 그 이유를 쓰고, 2) 서버 측·네트워크 측 대응방안을 각각 1가지 서술하시오.
모범답안 보기
1) monlist — 최근 접속 목록(최대 600건)을 소량 요청에 대용량으로 응답 → 큰 증폭률로 악용 2) · 서버 측: NTP를 4.2.8 이상으로 업그레이드 또는 disable monitor로 monlist 비활성화 · 네트워크 측: 방화벽에서 UDP 123 외부 트래픽 차단 / uRPF로 출발지 위조 필터링

스니핑 탐지

19회25회26회gap 6 · 주기 3.5
Promiscuous 모드 · 탐지 4법
  • 전제: NIC를 무차별(Promiscuous) 모드로 두면 목적지가 자신이 아닌 패킷도 모두 수신
  • Ping: 존재하지 않는 위조 MAC + 정상 IP로 ICMP → 응답하면 무차별 모드
  • ARP: 위조 MAC ARP 요청 → 응답 오면 스니퍼
  • DNS: 스니퍼의 역방향 DNS 조회 트래픽 감지 · Decoy: 가짜 계정 흘려 사용 관찰
  • 대응: SSL/SSH 암호화 통신, 지능형 스위치, ifconfig eth0 -promisc
📚 기출 · 26회 15번(서술)
로그 [device eth0 entered Promiscuous mode] 관련. 1) Promiscuous mode의 의미 2) 가능한 공격 3) 대응 방법
모범답안 보기
1) eth0로 들어오는 모든 패킷을 수신(목적지가 자신이 아니어도 drop하지 않음) 2) 패킷 스니핑(도청) 공격 3) ① SSL/SSH 암호화 통신 ② 지능형 스위치로 불필요 브로드캐스트 최소화 ③ ifconfig eth0 -promisc로 해제 ④ 스니핑 탐지도구 운영
GICHUL 26-15 · 19-13(동일 유형) · 25-14(ping 탐지)
📝 32회 예상문제
스니핑 탐지 방법 중 1) ARP를 이용한 방법과 2) Decoy(유인) 방법의 원리를 각각 서술하고, 3) 근본적 대응책 1가지를 쓰시오.
모범답안 보기
1) 존재하지 않는 위조 MAC으로 ARP 요청 → 정상 호스트는 무시하지만, 무차별 모드 스니퍼는 응답 → 스니퍼 식별 2) 가짜 ID/비밀번호를 평문으로 흘려(유인) 두고, 그 계정으로 접속 시도가 발생하면 스니핑 중인 공격자 존재를 탐지 3) SSL/SSH 등 암호화 통신 (스니핑돼도 내용 보호)

특수권한 SUID/SGID·파일권한

19회24회27회gap 5 · n3
특수비트 · 점검
  • SetUID 4000(실행 시 소유자 권한) · SetGID 2000(소유 그룹 권한) · Sticky 1000(디렉터리 삭제 제한, /tmp)
  • 표기: -rwsr-xr-x의 소유자 실행자리 s=SetUID, 그룹 자리 s=SetGID, 기타 실행자리 t=Sticky
  • 점검: find / -perm -4000 -type f · 제거: chmod -s 파일
  • 위험: SUID root 바이너리가 침해되면 권한 상승 → 불필요한 SUID 제거가 핵심
📚 기출 · 24회 13번(서술)
다음 조치가 필요한 이유를 설명하시오. 1) chmod -s {파일명} 2) find / -user root -type f \( -perm -4000 -o -perm -2000 \)
모범답안 보기
1) 파일의 특수비트(SetUID/SetGID)를 제거 2) / 하위에서 root 소유 + 특수비트 설정 파일을 검색 이유: 특수비트 파일 실행 시 소유자(root) 권한으로 동작 → 취약하면 권한 상승 공격의 통로가 되므로, 불필요한 SUID/SGID를 찾아 제거해야 함
GICHUL 24-13 · 19-11(권한 해석) · 27-15(rwx 해석)
📝 32회 예상문제
파일 권한이 -rwsr-xr-x root root /usr/bin/backup 이다. 1) 소유자 실행자리 s의 의미와 실행 시 동작을 서술하시오. 2) 이 설정이 위험한 이유와 점검·제거 명령을 쓰시오.
모범답안 보기
1) SetUID 설정 → 누가 실행하든 소유자 root 권한으로 동작 2) 취약한 SUID root 파일은 권한 상승 공격에 악용 가능. 점검: find / -perm -4000 -type f · 제거: chmod -s /usr/bin/backup
